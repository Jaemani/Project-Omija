# ADR-0008: Dashboard-First Demo Surface

Date: 2026-07-04
Status: Approved

## Context

The integrated HTML output originally read like a verification report: feed
counts, link checks, and generated artifact status. That proves the pipeline
works, but it does not immediately show the product value to a judge or analyst.

The demo needs to answer the operational question in the first viewport:

> Which supplier creates an active path to a protected program, why does it
> matter now, and what should be reviewed first?

## Decision

Make `out/intelligence_demo.html` a dashboard-first surface generated by
`scripts/intelligence_demo.py`.

Design guidance was requested from Claude Opus and implemented by Codex. The
dashboard uses a watch-floor console structure:

1. `Current Threat Picture` with operational value and top metrics.
2. `Ontology Path — Credential To Program` showing the active graph path.
3. `Public Source Corroboration` showing OSINT context.
4. `Recommended Response — Analyst Review Required` ranking actions.
5. `Unsent Advisory — Draft` showing human approval gating.
6. `Collection Status` separating public OSINT, Foundry OSDK, and credential
   feed status.

## Rationale

The product value is not "we fetched data." The value is the fused decision:

- exposed supplier identity;
- target asset access surface;
- supplier and prime graph traversal;
- impacted programs;
- public OSINT context;
- response draft that is not automatically sent.

This is more credible for defense and intelligence users than a marketing hero
or a raw CI report.

## Consequences

Positive:

- The first viewport now explains system purpose and value.
- The ontology path is visible as a decision chain.
- Feed degradation is shown as operational status, not hidden as failure.
- Human review remains explicit.

Tradeoffs:

- The static dashboard is still a generated artifact, not a full interactive
  Workshop app.
- Live credential ingestion remains dependent on external feed authentication.

## Guardrails

- Do not claim synthetic seed credentials are real leaked data.
- Do not show raw secrets, JWTs, cookies, or bearer tokens.
- Do not imply notification auto-send.
- Do not hide credential-feed auth degradation.
