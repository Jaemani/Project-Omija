# Palantir-style analyst pages — three empty-state versions (owner review)

Generated by `scripts/palantir_pages.py` (one generator → three files):

- `out/palantir_v1.html` — **V1 "Investigation"** (Gotham-like)
- `out/palantir_v2.html` — **V2 "Ops Triage"** (Workshop-like)
- `out/palantir_v3.html` — **V3 "Data Contracts"** (empty-slots-with-candidates, literal)

All three are **structure only**: every data slot renders its EMPTY state next to a
"data contract card" describing the candidate input shape with masked placeholders
(`•••`, `‹fqdn›`, `‹ISO-8601›`). No network, no ingestion, no synthetic credentials
or real org names, vendor-neutral language throughout. Object/link/state names are the
API identifiers (English); Korean is used only for explanatory captions.

The owner picks which one to keep. No recommendation here about data acquisition — that
is the owner's call. This doc records, per version, what it optimizes for, its
tradeoffs, which candidate feeds fill which slots, its **ontology leverage** (where the
ontology is used and why), and open questions.

---

## Schema metadata — how it was sourced (hybrid)

- **Live from the engine (imported, truthful).** Scoring bands / thresholds / weights
  from `actions.scoring.SCORING` (`active_floor=70`, `base_cap=60`,
  `grade_thresholds={즉시:70, 주의:40}`, exposure/recency/secret/module weights); the
  active-compromise window and privileged-account set from `actions.flag_active`
  (`ACTIVE_WINDOW_DAYS=14`, `{vpn, admin}`); program-rollup config from
  `actions.propagate_risk.PROGRAM_SCORING`; feed-module confidence tiers from
  `adapter.base.CONFIDENCE`; and the four refusal exception names
  (`EvidenceRequired`, `PathIncomplete`, `ProgramEvidenceRequired`, `CitationRequired`).
  These are read at generation time, so the pages cannot drift from the engine's actual
  guarantees.
- **Hardcoded (well-commented schema dict in the generator).** Object property shapes
  (`OBJECT_TYPES`), link list (`LINK_TYPES`), and lifecycle state names. Reason: the
  prose ontology docs are not reliably machine-parseable, and some lifecycle states are
  the **target Foundry model** — the current SQLite engine emits only the initial state
  (`open` for incidents, `draft` for drafts, `pending` for merge proposals), while
  `flagged→acknowledged→assigned→closed` and `draft→reviewed→approved→exported` are the
  human-on-the-loop transitions the ontology defines. A single commented dict is the
  honest source for that.

Feed-module codes are **abstracted to vendor-neutral semantic labels** (stealer-log,
url·login·pass, breach dump, combo list) so no feed vendor is named anywhere.

## The organizing problem (all three are built to answer it, even empty)

> Which supplier appears to create an **active** path into a protected defense program,
> why does it matter now, and what action should be prepared first?

### Invariants encoded visually in every version

1. **Triage ordering is lexicographic (band, score).** Band A = active compromise path
   established. Because the live `active_floor (70) > base_cap (60)`, one active path
   mathematically outranks any volume of non-active leaks. Bands shown as A/B/C lanes /
   badges, queues empty.
2. **`of` vs `targets` separation.** Whose credential (Identity's home org) vs what asset
   it appears to access (possibly a *different* org). Made visible in the V1 graph
   (curved orange `targets` cross-edge), the V1 path strip, and the V2 entity-360 note.
3. **Provenance mandatory.** Every derived-judgment slot (RiskAssessment,
   CompromiseIncident, ProgramExposure, NotificationDraft) shows an evidence drill-down
   affordance (`evidenced_by` / `cites`), empty but present, with the refusal exception
   named.
4. **Human-on-the-loop state machines.** Incident
   `flagged→acknowledged→assigned→closed(remediated|false_positive)`; NotificationDraft
   `draft→reviewed→approved→exported` — there is **no `sent` state**; the system never
   sends.

## V1 — "Investigation" (Gotham-like)

**Optimizes for:** *understanding the model*. Three columns — left object-type browser
(13 types, all counts 0, grouped by kind), center canvas that **renders the ontology
schema itself as the graph** (object types = nodes, link types = labelled edges; since
there is no data, the schema IS the graph), right inspector (property/link/lifecycle/
provenance sections, all empty, with a contract card). Clicking a type or a graph node
drives the inspector.

**Tradeoffs.** Strongest at "prove this is a graph, not a table," weakest at "what do I
*do* right now." The schema graph is dense — good for a technical judge, potentially
busy for an operator. Interaction is client-side JS (self-contained, no CDN).

**Which candidate feeds fill which slots.** CredentialExposure nodes ← credential-exposure
record; InfectedDevice ← infostealer device observation; Supplier/Domain ← supplier
registry row. Derived nodes (RiskAssessment/Incident/ProgramExposure) and output nodes
(MergeProposal/NotificationDraft) are produced by the pipeline, never ingested.

**Ontology leverage — where the ontology is used and why (answers on-page):**
- **Schema-as-graph canvas** — the highlighted credential→program chain is labelled with
  *why each hop exists* (`of` = whose credential; `belongs_to`/`owns⁻¹` = home org;
  `subcontractsTo*` = variable-depth tier chain; `supplies`/`runs` = reach to program).
  A flat table cannot express a variable-depth reachability path.
- **`of` vs `targets`** — the curved cross-edge shows the credential can *target* a
  different org's asset than it belongs *to*. This cross-org edge is what makes "active
  path into a program" expressible at all.
- **Inspector provenance section** — each derived object names its mandatory
  `evidenced_by`/`cites` link and the exception raised if empty. Auditability is a
  property of the model, shown even with zero rows.
- **Inspector lifecycle section** — actions render as object state machines, surfacing
  human-on-the-loop as a model property.

## V2 — "Ops Triage" (Workshop-like)

**Optimizes for:** *operating the queue*. A band-ordered triage queue (A/B/C lanes,
empty, each lane states the evidence shape that would fill it and the path/record shape
it implies), an entity-360 template for a single supplier, the CompromiseIncident state
machine, and a NotificationDraft workspace (empty editor, `cites`-required notice,
approve/export-only state row, prominent **NO SEND**).

**Tradeoffs.** Strongest at "what do I do first and why now," and it makes the
active-on-top invariant explicit with the live `active_floor > base_cap` argument.
Weaker at showing the full object model (it foregrounds the three or four objects an
operator touches). Least dense of the three.

**Which candidate feeds fill which slots.** Band A entries require an active
CompromiseIncident (needs infostealer device observation + credential-exposure record
correlated to a supplier registry row). Band B/C entries require only correlated
credential-exposure records (no active path). Entity-360 slots are filled by the same
three feeds plus pipeline-derived objects.

**Ontology leverage — where the ontology is used and why (answers on-page):**
- **Band queue caption** — bands are captioned as a *graph consequence*: `active_flag`
  comes from path existence, and `active_floor(70) > base_cap(60)` guarantees active
  outranks stale volume. The invariant is derived from the graph, not tuned by hand.
- **Entity-360** — annotated with the `of`/`targets` separation: a supplier's 360 view
  is only correct if credential ownership and asset target are distinct edges.
- **Incident state machine** — human-on-the-loop is a model property; the app never
  auto-transitions.
- **Draft workspace** — `cites` mandatory (else `CitationRequired`); no `sent` state
  exists in the object model, so "the system never sends" is structural, not a policy in
  app code.

## V3 — "Data Contracts" (empty-slots-with-candidates, literal)

**Optimizes for:** *making the empty-slots idea literal*. Three sections: (1) per-object
contract cards for all 13 types (key properties + types + masked placeholders +
lifecycle + required provenance link); (2) the pipeline
`correlate → flag_active → compute_risk → propagate → notify_draft` as a flow with empty
counters and per-stage input/output contracts **and the refusal condition** for each
stage; (3) a "candidate feed shapes" section with three vendor-neutral JSON-shape cards
(credential-exposure record, infostealer device observation, supplier registry row),
values masked, each tagged with the object-type slot it fills.

**Tradeoffs.** Strongest at "here is exactly what data we need and what each piece
becomes," directly serving the owner's empty-slots-with-candidates requirement. It is the
most catalog-like (least "console"), so it reads as a spec/contract sheet more than a live
tool. Best companion to a data-onboarding conversation.

**Which candidate feeds fill which slots (this is the explicit content of the page).**
- credential-exposure record → CredentialExposure (`of` Identity, `targets` Domain,
  `sourced_from` ThreatSource).
- infostealer device observation → InfectedDevice (`leaked` Exposure, `compromises`
  Identity; carries the active-compromise discriminators cookie / account_type /
  infected_at).
- supplier registry row → Supplier + Domain (+ `subcontractsTo` edges for the tier chain).

**Ontology leverage — where the ontology is used and why (answers on-page):**
- **Contract cards** — every derived card carries its mandatory provenance link and the
  refusal exception, so provenance-mandatory is visible per object.
- **Pipeline refusal conditions** — each stage names the exception it raises when the
  required links are missing (`EvidenceRequired`, `PathIncomplete`,
  `ProgramEvidenceRequired`, `CitationRequired`). The graph enforces auditability; the
  pipeline cannot manufacture a derived object without evidence.
- **Feed-shape → slot mapping** — shows that a raw feed row carries *both* an identity and
  a host, which the ontology splits into `of` (owner) and `targets` (asset); the
  cross-org edge only exists because these are separate links.
- **PropagateRisk rollup note** — a Program's exposure is a multi-hop aggregate over all
  reaching supplier paths, deduped by distinct Supplier (diamond supply chains are not
  double-counted) — only a path graph gets this right.

## Version comparison (one line each)

| | optimizes for | density | best judge | weakest at |
|---|---|---|---|---|
| **V1 Investigation** | understanding the model (schema-as-graph) | high | technical / architecture | "what do I do now" |
| **V2 Ops Triage** | operating the queue, active-on-top | medium | operator / CISO | full object-model coverage |
| **V3 Data Contracts** | what data we need + what it becomes | catalog | data onboarding / integrator | reads as a spec, not a console |

They are not mutually exclusive: V1's schema graph, V2's band queue, and V3's contract
cards could compose into one console later. Kept separate here so the owner can judge each
framing on its own.

## Design / accessibility notes

- Dark, dense, Palantir-esque; committed single dark look (not theme-toggling — these open
  as static files). Near-black plane, thin hairlines, mono API labels, restrained accents.
- Palette drawn from the dataviz reference (dark column). Categorical schema-graph hues
  `{blue #3987e5, aqua #199e70, yellow #c98500, violet #9085e9}` **pass the validator** on
  the dark surface `#1a1a19` (lightness band, chroma floor, CVD separation, contrast).
  Triage bands use the reserved status ramp (critical / warning / info) with A/B/C letter
  badges + labels as secondary encoding — status colors are never reused as a series hue
  (an earlier draft did; fixed).
- No horizontal page scroll; wide content (schema SVG, pipeline row, contract grid)
  scrolls inside its own `overflow-x` container.
- A live-browser visual pass was not run (the QA browser needs a one-time build and the
  task did not require it); structure was verified by grep/heading checks. Opening the
  three files in a browser is the remaining manual eyeball, mainly for SVG label spacing
  in V1.

## Open questions for the owner

1. **One console or three framings?** Keep one, or compose V1 graph + V2 queue + V3
   contracts into a single tabbed console?
2. **Lifecycle states — target vs current.** The pages show the full target lifecycle
   (`flagged→acknowledged→assigned→closed`, `draft→reviewed→approved→exported`). The
   engine currently persists only the initial state. Should the pages annotate which
   transitions are implemented vs planned, or is the target model the right thing to show
   a judge?
3. **`Prime` as object vs `Supplier(is_prime, tier=0)`.** The v0.3 ADR candidate. If Prime
   folds into Supplier, the schema graph and contract cards change. Show both, or commit?
4. **Band thresholds on-page.** V2 surfaces the live `active_floor`/`base_cap` numbers.
   Keep the exact numbers visible to a judge, or abstract to "active band / base band"?
5. **Domain asset detail.** Contract cards include `asset_type` / `access_surface`
   (vpn/sso/admin…). Is that the right granularity for the `targets` asset, or should the
   asset side stay a plain host string until the registry is richer?
