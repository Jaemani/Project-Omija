# Steady-State Console — 방향 문서 (Fable, 2026-07-05)

구현 주체: Codex. 이 문서는 방향과 요건만 정의한다. 민감정보 열람 구역은 **비워둔 채** 계약(요건)만 명시한다.

## 1. 목적

사건이 없을 때 분석가가 기본으로 켜놓는 화면. 두 가지를 증명한다:

1. **감시 태세** — 무엇을 얼마나 덮고 있는가 (coverage)
2. **시스템 신뢰** — 조용한 것이 "안 보이는 것"이 아니라 "봤는데 없는 것"임 (negative evidence)

사건 보고서(omija_demo.html)가 "왜 지금 움직여야 하나"라면, 이 화면은 "평소에 뭘 하고 있나". 둘이 합쳐져야 운영 시스템이지 보고서 생성기가 아니다.

## 2. 패널 구성 (우선순위 순)

### P1. 감시 범위 (coverage map)
- 등록 협력사 n, 감시 도메인 n, 연결 프로그램 n — registry 실측 (가상 라벨)
- 티어 분포 미니맵: prime → tier-1 → tier-2 트리 요약, 각 노드에 감시 상태 뱃지
- 공백 표시가 핵심: "등록됐지만 도메인 미확인 협력사 k개" — 커버리지의 구멍을 숨기지 않고 보여주는 것 자체가 신뢰 장치

### P2. 조용함의 증명 (all-clear evidence)
- "활성 경로 0 — 마지막 전수 평가 {timestamp}, 평가 대상 {n}개 경로, 다음 주기 {t}"
- 최근 평가에서 기각된 후보 수 (band C로 분류된 것들) — "0건 탐지"가 아니라 "n건 검토, 0건 활성"
- 엔진 불변식 상태 표시: active_floor > base_cap 등 — 시스템 자기 검증

### P3. 피드 상태 (collection posture)
- credential exposure feed: **비활성 / 승인 대기** — 빈 슬롯, 계약 카드만 (벤더 중립)
- OSINT feed: 상태 표시만
- Foundry sync: 마지막 readback 시각, 온톨로지 버전
- 정직한 상태 표시가 목적. "연결된 척" 금지.

### P4. 결정 감사 스트림 (live)
- Foundry 액션 감사로그 최근 항목: 누가 언제 어떤 상태 전이를 실행했나
- out/foundry_action_chain.json 증적 재사용 가능
- 이 패널이 "human-on-the-loop이 실제로 작동 중" 증거

### P5. 민감정보 열람 구역 — **비워둠, Codex 구현**
UI는 잠긴 상태의 자리만 만든다. 요건 계약:

- **접근 게이트**: 열람 사유 입력 필수 → 사유가 감사로그에 기록된 후 개방
- **마스킹 기본값**: 비밀값은 항상 `•••` + fingerprint. 원문 열람은 별도 권한 + 별도 감사 이벤트
- **세션 표시**: 열람자 ID·시각 워터마크
- **열람 범위**: 레코드 단위 (전체 덤프 열람 없음)
- **금지**: 원문 저장·복사 UI, 일괄 export
- 슬롯 문구 예시: "이 구역은 승인된 피드 연결 및 열람 권한 정책 확정 후 활성화됩니다"

## 3. 데이터 출처 규율

모든 표시값에 provenance 칩 필수 (omija_demo와 동일 체계):
`LIVE·Foundry` / `ENGINE·실측` / `SEED·가상` / `FRAME·연출`

- 자동 조치·발송류 UI 없음
