## D. 깊이 무관성 증명 (depth-agnosticism)
**맥락**: v0.2의 재귀 traverse는 가변깊이를 처리하지만, 데모가 항상 깊이 2(2차→1차→원청)만 보이면 심사관이 "고정 조인 2번을 재귀로 포장"이라 깎을 수 있음(구조검토 §5 급소).

**제안 내용**:
- `adapter/mock.py`/`registry/suppliers.yaml`에 **깊이 3 체인**(3차→2차→1차→원청→프로그램) 1개 추가.
- 피칭/`docs/demo.md`에 "동일한 `WITH RECURSIVE` 질의 하나가 깊이 1·2·3을 모두 처리 — flat join으로 불가능"을 명시.

**왜 지금 안 하나**: v0.2 기본 전파가 아직 착지 전. 기본기부터 견고히 한 뒤 데모 보고 필요성 판단(유저 "억지 금지" 라인).

**예상 규모**: 소(mock 1체인 + 데모 문구 + 테스트 1). 기존 재귀 질의 재사용이라 로직 변경 없음.

**스멜테스트 함의**: §1(다중홉) 강화. 새 객체·링크 없음 → 스코프 안전.

## E. ProgramExposure 액션 라이프사이클
**맥락**: 현재 설계의 ProgramExposure는 계산된 수치(수동적 롤업). 스멜테스트 §3(액션=상태전이)에 대해 다른 파생객체(RiskAssessment/Incident/Draft)보다 약함 — §1(다중홉 집계)·§4(provenance)로는 강하게 통과하지만.

**제안 내용**:
- ProgramExposure에 상태 필드 + 액션: `open → acknowledged → escalated`. CISO가 "프로그램 X 타는 중" 승인·에스컬레이션(프로그램 보안책임자에게).
- 효과: "타는 프로그램"이 단순 순위 숫자가 아니라 **결정 대상 객체**가 됨 → 스멜테스트 §3 강하게 통과 + 프로그램 단위 에스컬레이션 = 심사 "군 배치성 30%" 직결.

**왜 지금 안 하나**: 객체에 라이프사이클 추가는 스코프 확장. 데모에서 "프로그램 순위"만으로 충분히 임팩트 나는지 먼저 확인(과설계 방지).

**예상 규모**: 중(상태 필드 + 액션 2개 + 스토어 + 대시보드 상태 UI + 테스트).

**스멜테스트 함의**: §3 강화. 단, ProgramExposure는 이미 §1·§4로 통과하므로 이 승격은 "정당성 확보"가 아니라 "배치성 강화" 목적 — 억지 아님을 재확인할 것.

## 참고: 스코프 규율 (ontology.md §5)
객체·링크·라이프사이클 추가 전 스멜테스트 §0 재통과 필수. 통과 못 하면 속성이지 객체 아님. 특히 **중간 경유 Supplier(tier-1)용 별도 파생 위험 객체는 만들지 않기로** 확정 — 경로 질의로 유도 가능 + 상태전이 없음 = 탈락. 경유 업체는 ProgramExposure.components와 incident path 표시로만.
